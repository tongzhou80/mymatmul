# 一步步优化 GEMM：从朴素实现到高性能计算

## 背景：数据类型与浮点数表示

在深入 GEMM 优化之前，理解数据类型（尤其是浮点数格式）非常重要，因为它们会直接影响计算性能和数值稳定性。

### 浮点数在计算机中的表示方式

现代计算机通常遵循 IEEE 754 标准。一个浮点数由三部分组成：
- **符号位（Sign, S）**：1 bit，表示正负
- **指数（Exponent, E）**：决定数值范围（scale）
- **尾数 / 有效位（Mantissa, M）**：决定精度

其表示形式为：

$(-1)^S × 1.M × 2^(E - bias)$


可以直观理解为：
- 指数位越多 → 表示范围（dynamic range）越大
- 尾数位越多 → 精度（precision）越高

这两者之间本质上是一个 **trade-off**。

### FP32：经典基准

- **1 位符号**
- **8 位指数**
- **23 位尾数**

FP32 在范围和精度之间取得了较好的平衡，因此长期以来是科学计算和深度学习的默认选择。

### FP16：高性能但范围有限

- **1 位符号**
- **5 位指数**
- **10 位尾数**

**FP16 的优势**：
- 更小的存储（节省一半内存）
- 更高的计算吞吐（尤其是在 GPU Tensor Core 上）

**但问题也很明显**：
- 指数位太少（5 bit） → 动态范围很小
- 容易出现：
  - 梯度下溢（underflow）
  - 数值溢出（overflow）

因此在训练中往往需要额外技巧（例如 loss scaling）。

### bfloat16：保留范围，牺牲精度

- **1 位符号**
- **8 位指数**（和 FP32 一样）
- **7 位尾数**

**关键设计**：
👉 **直接继承 FP32 的指数范围**
👉 **压缩尾数精度来换取更小的存储和更高性能**

### FP16 vs bfloat16 数值范围对比

#### FP16 的范围（normalized 数）

FP16 的指数 bias 为：$bias = 2^{5-1} - 1 = 15$

对于 normalized 数，真实指数范围为：$e \in [-14, 15]$

- **最小 normalized 正数**: $x_{min}^{FP16} = 2^{-14} \approx 6.1 \times 10^{-5}$
- **最大 normalized 正数**: $x_{max}^{FP16} = (2 - 2^{-10}) \times 2^{15} = 65504$

#### bfloat16 的范围（normalized 数）

bfloat16 的指数 bias 为：$bias = 2^{8-1} - 1 = 127$

对于 normalized 数，真实指数范围为：$e \in [-126, 127]$

- **最小 normalized 正数**: $x_{min}^{BF16} = 2^{-126} \approx 1.18 \times 10^{-38}$
- **最大 normalized 正数**: $x_{max}^{BF16} = (2 - 2^{-7}) \times 2^{127} \approx 3.39 \times 10^{38}$

#### 对比总结

| 方面 | FP16 | bfloat16 |
|------|------|---------|
| 尾数位 | 10 bit | 7 bit |
| 指数位 | 5 bit | 8 bit |
| 最小值 | $6.1 \times 10^{-5}$ | $1.18 \times 10^{-38}$ |
| 最大值 | $65504$ | $3.39 \times 10^{38}$ |
| **精度** | **更高** | 较低 |
| **动态范围** | 有限 | **远大于 FP16** |

### 为什么深度学习训练更偏向 bfloat16？

在机器学习训练中，**"范围"往往比"精度"更重要**：

1. **梯度分布范围极广**
   - 梯度可能跨越多个数量级
   - FP16 很容易把小梯度直接变成 0（underflow）
   - bfloat16 可以保留这些信息

2. **神经网络对精度不敏感**
   - SGD 本身就是噪声优化
   - 少量 rounding error 通常不会影响收敛

3. **训练更简单**
   - FP16 通常需要：loss scaling、careful tuning
   - bfloat16 通常：开箱即用（out-of-the-box）

4. **硬件支持成熟**
   - TPU 从一开始就围绕 bfloat16 设计
   - NVIDIA A100 / H100 等 GPU 提供高效 bfloat16 Tensor Core
   - 性能接近 FP16，但稳定性更好

### 对 GEMM 优化的影响

数据类型直接影响 GEMM 的性能特征：

- **更低精度（FP16 / bfloat16）**
  - 降低内存带宽压力
  - 提高 cache 利用率
  - 提升计算吞吐

- **bfloat16 的优势**
  - 保持数值稳定性
  - 同时可以使用高吞吐的矩阵乘加单元

**实际工程中通常采用**：
👉 **输入**：bfloat16
👉 **累加**：FP32（accumulate in FP32）

这样可以在性能和精度之间取得良好平衡。

## 版本 1：朴素实现 - cuda_naive_ijk

### 设计原理

```cuda
int i = blockIdx.x * blockDim.x + threadIdx.x;  // i -> blockIdx.x
int j = blockIdx.y * blockDim.y + threadIdx.y;  // j -> blockIdx.y

if (i < M && j < N) {
    float sum = 0.0f;
    for (int k = 0; k < K; k++) {
        sum += A[i*K + k] * B[k*N + j];
    }
    C[i*N + j] = sum;
}
```

**核心思想**：
- 每个线程负责计算矩阵 C 中的一个元素
- 2D 线程网格：(i,j) 映射到 (blockIdx.x, blockIdx.y)
- 每个线程遍历整个 K 维度，直接访问全局内存

**特点**：
- 代码简洁，易于理解
- **全依赖全局内存**，带宽利用率差
- 无内存重用，每个乘法都是一次全局内存读取

### 性能表现

|---------|--------|---------|
| 64³     | 22.0   | 0.026   |
| 128³    | 160.3  | 0.027   |
| 256³    | 705.3  | 0.049   |
| 512³    | 1,062.8| 0.253   |
| 1024³   | 1,148.0| 1.872   |
| 2048³   | 1,163.4| 16.745  |

**观察**：
- 性能在 512³ 后**几乎完全停滞**
- 达到理论峰值的仅 **0.78%**（1,280 / 165,200）
- 线程映射与数据布局严重失配

## 版本 2：线程映射优化 - cuda_naive_ijk_jx

### 优化原理

关键洞察：**CUDA 中 x 维度变化最快，应与数据最快变化维度对齐**

```cuda
int j = blockIdx.x * blockDim.x + threadIdx.x;  // j -> blockIdx.x（x 最快）
int i = blockIdx.y * blockDim.y + threadIdx.y;  // i -> blockIdx.y

// 同样的计算，但线程映射调整
```

**为什么这很重要**：

在 C 的行优先存储中，`C[i*N + j]` 意味着 j 是最快变化的维度。

- **错误映射**（版本 1）：i 映射到 blockIdx.x，导致相邻线程访问的内存相隔 K 位置
- **正确映射**（版本 2）：j 映射到 blockIdx.x，相邻线程现在访问相邻内存


这种对齐使 **内存合并**（memory coalescing）成为可能——GPU 可以在一次事务中加载相邻的内存。

### 性能表现

| 矩阵规模 | GFLOPS | 提升倍数 | 时间(ms) |
|---------|--------|---------|---------|
| 64³     | 19.6   | 0.89x   | 0.031   |
| 128³    | 176.1  | **1.10x**   | 0.026   |
| 256³    | **1,264.3** | **1.79x** | 0.030   |
| 512³    | 3,326.2| **3.13x** | 0.108   |
| 1024³   | **5,128.2** | **4.47x** | 0.404   |
| 2048³   | **5,488.9** | **4.72x** | 3.073   |
| 4096³   | **5,429.8** | **4.24x** | 25.5    |

**关键观察**：
- **4.3 倍性能提升**（从 1,280 到 5,430 GFLOPS）
- 仅通过**调整线程映射**，不改变计算逻辑
- 这演示了**内存合并的威力**
- 现在达到理论峰值的 **3.3%**

### 优化要点

✓ **对齐 CUDA 线程维度与数据布局**
✓ **启用内存合并**（多个线程加载连续内存）
✓ **零额外开销**（无新数据结构或同步）

## 版本 3：共享内存分块 - cuda_tiled_32x32

### 优化原理：局部性和重用

朴素实现的根本问题：**零数据重用**

对于计算 `C[i,j] = Σ A[i,k] * B[k,j]`：
- A[i,k] 从全局内存读取 **N 次**（对每个 j）
- B[k,j] 从全局内存读取 **M 次**（对每个 i）
- 总读取次数：M×N×(M+N)

**分块策略**：将 K 维度分成 32 元素的瓦片，每个瓦片：
1. 加载 32×32 的 A 瓦片到共享内存
2. 加载 32×32 的 B 瓦片到共享内存
3. 计算这个瓦片对 C 的贡献

```cuda
for (int k_tile = 0; k_tile < K; k_tile += TILE_SIZE) {
    // 步骤 1：加载 A[i, k_tile:k_tile+32] 到共享内存
    A_shared[threadIdx.y][threadIdx.x] = A[i*K + (k_tile + threadIdx.x)];
    
    // 步骤 2：加载 B[k_tile:k_tile+32, j] 到共享内存
    B_shared[threadIdx.y][threadIdx.x] = B[(k_tile + threadIdx.y)*N + j];
    
    __syncthreads();  // 确保所有数据加载完成
    
    // 步骤 3：在共享内存中累积
    for (int kk = 0; kk < TILE_SIZE; kk++) {
        sum += A_shared[threadIdx.y][kk] * B_shared[kk][threadIdx.x];
    }
    
    __syncthreads();
}
```

**数据流改进**：
- A[i,k] 加载一次到共享内存，被该块的所有 j 线程重用
- B[k,j] 加载一次到共享内存，被该块的所有 i 线程重用
- **共享内存带宽 >> 全局内存带宽**
